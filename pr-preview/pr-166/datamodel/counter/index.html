Counter

Counter is a very special type that is specific to CRDTs. While they look like the number type, they are geared towards building applications where various different devices need to increment or decrement at the same time while preserving consistency. The most common use is to build a voting system or an inventory application. Building applications that needs a consistent count with only using the default number type will not be appropriate. This is where the counter comes in.

Counters can be changed through a special method called increment which takes a number to increment the counter by. If you wish to decrement the counter then you can supply a negative number.

To create a counter, first insert a document with number value. You must then call an update function to convert the number into a counter with the replaceWithCounter method. This will convert the number into a counter.

Once the value in the document is a counter, you can proceed to increment or decrement the value. This will preserve an accurate value once devices sync.

const frankId = await ditto.store.collection('people').upsert({  name: 'Frank',  ownedCars: 0, // here 0 is a number})
await ditto.store  .collection('people')  .findByID(frankId)  .update((mutableDoc) => {    mutableDoc._replaceWithCounterAt('ownedCars')    mutableDoc._incrementCounterAt('ownedCars', 1)  })
let docId = try! ditto.store["people"].upsert([    "name": "Frank",    "ownedCars": 0 // here 0 is a number])
ditto.store["people"].findByID(docId).update({ mutableDoc in    mutableDoc?["ownedCars"].replaceWithCounter()    mutableDoc?["ownedCars"].increment(amount: 1)})

When do I use a counter?​

Let's look through a typical example for a counter: four flight attendants walk through an airlane and record passenger meal orders. For each seat, they count the number of drinks ordered by passengers and add each item onto an ordered list. They also maintain a number of the total amount of drinks available for that type.

First, the flight attendant selects the flight they are on. This then renders an interface with the proper number of seats based on the plane type and flight manifest. This might be retrieved by an external API, or it could be internal to the app based on some known information about each plane type.

Each seat has a few values that the attendant can modify. Once each flight attendant is done walking through the airplane, they syncronize their devices together and have the full picture of how many drinks each passenger has ordered.

Code example​

First, when the app loads we call populateSeats(), and then we renderSeatList().

MainViewController.swift

This is where we populate all of the seats. We never want to block user input. So we need to make sure that each seat is created locally, even if there is no access to the internet or other devices in the network.

When we call replaceWithCounter(), the counter is initialized to 0. totalAvailable is a Register value that doesn't change unless we update it later, and usedCount which is a counter that starts at 0 and is incremented over a period of time.

seatIds.forEach {  try! collection.upsert([      "_id": $0,      "name": "Soda",      "usedCount": 0,      "totalAvailable": 100  ], writeStrategy: .insertDefaultIfAbsent)
  collection.findByID($0).update { newSeats in      newSeats?["drinks"].replaceWithCounter(isDefault: true)  }}

SeatFormViewController.swift

Then, we can render the seat list. We know that all of these documents have counters already, because when we selected this flight, we made sure that all of the seat documents existed.

We increment the counter when a user clicks the Plus button and decrement the counter when the user clicks the Minus button:

...set up the button}.onPlusDidClick {    seatDocument.update { doc in        doc?["drinks"].increment(amount: 1)    }}.onMinusDidClick {    seatDocument.update { doc in        doc?["drinks"].increment(amount: -1)    }}

Upserting a counter​

You should try to not use upsert to increment a counter. The upsert will turn the counter back into a register number. If another device then calls replaceWithCounter again to turn it back into a counter, all previous upsert calls will be treated as though they were incremented.

Use upsert to initialize the counter, but later use update for incrementing the counter.

Resetting counters​

After using a counter, you can't reset it back to 0. This is because counters have no knowledge of time. A counter "remembers" all previous increments. Each counter value is composed of the previous counter increment calls. This is true even if you change the type of a counter to another field and then back again.

If you need to reset a counter field to 0, you need to remove the counter from the document first.

let docID = try! ditto.store["drinks"].upsert([    usedCount: 10,])
ditto.store["drinks"].findByID(docID).update { mutableDoc in    mutableDoc?["usedCount"].replaceWithCounter()    mutableDoc?["usedCount"].increment(amount: 1)}
print(ditto.store["drinks"].findByID(docID)) // 11
ditto.store["drinks"].findByID(docID).update { mutableDoc in    mutableDoc?["usedCount"].increment(amount: 10)}
print(ditto.store["drinks"].findByID(docID)) // 21
ditto.store["drinks"].findByID(docID).update { mutableDoc in    mutableDoc?["usedCount"].remove()    mutableDoc?["usedCount"] = 0     mutableDoc?["usedCount"].replaceWithCounter()}
print(ditto.store["drinks"].findByID(docID)) // 0
ditto.store["drinks"].findByID(docID).update { mutableDoc in    mutableDoc?["usedCount"].increment(amount: 1)}
print(ditto.store["drinks"].findByID(docID)) // 1

Rationale​

We never want to block user input, so we populate all of the known data that a user might interact with on the first load of the flight. This means the flight attendant can start collecting information from scratch even if they have no connections to other peers.
